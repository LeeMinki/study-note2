## Clarifications

### Session 2026-04-16

- Q: 이번 MVP의 기본 배포 토폴로지는 무엇인가? → A: 단일 EC2에서 k3s, 선언적 배포 관리 도구, 애플리케이션을 함께 운영하는 저비용 MVP로 확정한다.
- Q: 선언적 배포 관리 도구는 어떤 운영 형태를 우선 검토하는가? → A: 더 가벼운 구성을 우선 검토하고, UI가 필수가 아니면 core 중심 구성을 기본안으로 비교한다.
- Q: 외부 접속 경로는 어떻게 제한하는가? → A: 하나의 공인 엔드포인트만 우선 제공한다.
- Q: 도메인과 HTTPS의 우선순위는 어떻게 두는가? → A: 이번 MVP에서는 선택사항으로 두고, 외부 접속 가능성 확보와 비용 영향 기준으로 우선순위를 비교한다.
- Q: 배포 자동화 흐름은 어떻게 나누는가? → A: PR 체크와 main 병합 후 자동 배포를 분리하고, PR 시점에는 프로덕션 배포를 하지 않는다.
- Q: 초기 필수 검증 항목은 무엇인가? → A: 테스트 코드 대신 인프라 포맷·검증, 앱 빌드, 컨테이너 이미지 빌드, 배포 명세 기본 검사를 중심으로 설계한다.
- Q: AWS 인증 방식의 기본안은 무엇인가? → A: 장기 액세스 키 대신 GitHub Actions OIDC 우선으로 설계한다.
- Q: 이미지 저장소와 운영 가시성은 어떻게 다루는가? → A: 컨테이너 이미지 저장소는 비용과 단순성 기준으로 선택안을 비교하고, 로깅·모니터링은 최소 수준만 다루며 본격 도입은 후속 spec으로 남긴다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Public MVP Environment (Priority: P1)

운영자는 아직 준비되지 않은 AWS 계정 상태에서 최소 리소스만으로 Study Note를 배포하고, 외부 사용자가 실제로 접속 가능한 단일 운영 환경을 확보할 수 있어야 한다.

**Why this priority**: 실제로 접속 가능한 배포 환경이 이번 기능의 핵심 목적이며, 이 흐름이 없으면 나머지 배포 자동화나 운영 문서도 가치가 없다.

**Independent Test**: 빈 AWS 환경에서 제공된 인프라 정의와 부트스트랩 절차만 따라 단일 서버 환경을 만들고, 외부 네트워크에서 Study Note 접속 가능 여부를 확인하면 된다.

**Acceptance Scenarios**:

1. **Given** AWS에 관련 리소스가 없는 상태에서, **When** 운영자가 정의된 인프라 절차를 실행하면, **Then** 단일 서버 기반의 배포 환경이 생성되어야 한다.
2. **Given** 배포 환경이 준비된 상태에서, **When** 외부 사용자가 공개된 접속 경로로 접근하면, **Then** Study Note 애플리케이션에 접속할 수 있어야 한다.

---

### User Story 2 - Repeatable App Release Flow (Priority: P2)

운영자는 애플리케이션 배포 구성을 선언적으로 관리하고, 동일한 절차로 초기 배포와 이후 재배포를 반복할 수 있어야 한다.

**Why this priority**: MVP라도 수동 서버 수정에 의존하면 재현성과 유지보수성이 급격히 떨어지므로, 배포 구성과 애플리케이션 구성이 반복 가능해야 한다.

**Independent Test**: 기존 환경이 준비된 상태에서 배포 구성을 다시 적용해도 수동 서버 조작 없이 애플리케이션이 원하는 상태로 유지 또는 갱신되는지 확인하면 된다.

**Acceptance Scenarios**:

1. **Given** 인프라가 이미 준비된 상태에서, **When** 운영자가 배포 구성을 다시 적용하면, **Then** 애플리케이션은 동일한 방식으로 재배포 가능해야 한다.
2. **Given** 배포에 필요한 애플리케이션 설정값이 준비된 상태에서, **When** 운영자가 배포를 수행하면, **Then** 민감한 값은 코드와 분리된 방식으로 적용되어야 한다.
3. **Given** 변경사항이 검토 중인 상태에서, **When** PR이 열리면, **Then** 배포 전 검증 흐름만 실행되고 프로덕션 배포는 실행되지 않아야 한다.
4. **Given** 변경사항이 main에 병합된 상태에서, **When** 운영 배포 흐름이 시작되면, **Then** 동일한 선언적 절차로 운영 환경에 자동 반영될 수 있어야 한다.

---

### User Story 3 - Minimal Operations Runbook (Priority: P3)

운영자는 문서만 보고도 배포, 설정값 반영, 기본 점검, 장애 시 1차 대응 절차를 따라갈 수 있어야 한다.

**Why this priority**: 테스트 코드와 고급 운영 스택이 없는 MVP 단계에서는 문서 품질이 운영 안정성을 직접 좌우한다.

**Independent Test**: 배포 경험이 없는 팀원이 운영 문서만으로 초기 배포, 재배포, 접속 확인, 기본 복구 단계를 수행할 수 있으면 된다.

**Acceptance Scenarios**:

1. **Given** 운영 문서가 준비된 상태에서, **When** 다른 팀원이 문서만 따라 배포를 시도하면, **Then** 주요 단계와 필요한 입력값을 빠짐없이 확인할 수 있어야 한다.
2. **Given** 서비스가 비정상 상태가 된 경우, **When** 운영자가 문서를 확인하면, **Then** 최소한의 점검 순서와 복구 순서를 확인할 수 있어야 한다.

### Edge Cases

- 인프라 생성이 일부만 완료된 상태에서 다시 실행해도 중복 생성이나 수동 정리 의존 없이 이어서 진행할 수 있어야 한다.
- 서버 재부팅 또는 인스턴스 재생성 후에도 애플리케이션을 다시 복구하는 절차가 문서화되어 있어야 한다.
- 공개 도메인이 아직 없는 경우에도 임시 접속 경로로 외부 검증이 가능해야 한다.
- 도메인과 HTTPS를 아직 적용하지 않은 경우에도 외부 접속 가능 여부를 우선 검증할 수 있어야 하며, 적용 여부 판단 기준이 문서화되어 있어야 한다.
- 필수 환경변수 또는 시크릿이 누락된 경우 배포 실패 원인을 확인할 수 있어야 한다.
- 비용 절감을 위해 단일 서버를 사용하더라도 디스크 부족, 공인 IP 변경, 수동 재배포 같은 운영 제한 사항이 명시되어 있어야 한다.
- PR 검증은 통과했지만 main 병합 후 배포가 실패한 경우, 운영자가 어디서 실패했는지 구분할 수 있어야 한다.
- 선택한 이미지 저장소가 일시적으로 사용할 수 없는 경우에도 대체 복구 절차나 재시도 기준이 문서화되어 있어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 AWS에 최소 비용의 단일 운영 환경을 새로 구성할 수 있어야 한다.
- **FR-002**: 시스템은 인프라 구성을 선언적으로 관리하여 동일한 입력으로 반복 적용할 수 있어야 한다.
- **FR-003**: 시스템은 단일 가상 서버에서 컨테이너 워크로드를 실행할 수 있는 클러스터 환경을 자동으로 준비해야 한다.
- **FR-004**: 시스템은 같은 서버 환경 안에 애플리케이션 배포 상태를 관리할 수 있는 선언적 배포 관리 도구를 제공해야 한다.
- **FR-004a**: 시스템은 이번 MVP에서 더 가벼운 운영 구성을 우선 검토해야 하며, UI가 필수적이지 않다면 경량 배포 관리 구성을 기본안으로 다뤄야 한다.
- **FR-005**: 시스템은 Study Note 애플리케이션을 클러스터 환경에 배포할 수 있는 배포 명세 구조를 제공해야 한다.
- **FR-006**: 시스템은 외부 사용자에게 노출되는 접속 경로를 정의하고, 초기 검증 시 실제 접속 가능 상태를 확인할 수 있어야 한다.
- **FR-006a**: 시스템은 이번 MVP에서 하나의 공인 엔드포인트만 제공해야 한다.
- **FR-006b**: 시스템은 도메인과 HTTPS 적용 여부를 선택사항으로 두되, 비용, 복잡도, 초기 검증 편의성 관점의 비교 기준을 문서화해야 한다.
- **FR-007**: 시스템은 애플리케이션 설정값과 민감 정보를 코드 저장소와 분리된 방식으로 주입할 수 있어야 한다.
- **FR-007a**: 시스템은 배포 자동화에서 장기 액세스 키 대신 단기 신뢰 기반 인증 방식을 우선 사용해야 한다.
- **FR-008**: 시스템은 초기 인프라 준비, 서버 부트스트랩, 애플리케이션 배포, 재배포 순서를 문서로 제공해야 한다.
- **FR-009**: 시스템은 기본 운영 문서에 접속 확인, 로그 확인, 재시작, 재배포, 장애 시 1차 대응 절차를 포함해야 한다.
- **FR-009a**: 시스템은 로깅과 모니터링을 최소 운영 수준으로만 다루고, 본격적인 수집·시각화·알림 체계는 후속 범위로 남겨야 한다.
- **FR-010**: 시스템은 이번 MVP에서 비용을 낮게 유지하기 위해 단일 서버, 단일 클러스터, 단일 환경만을 대상으로 해야 한다.
- **FR-011**: 시스템은 고가용성, 다중 가용 영역, 다중 노드, 본격적인 관측성 스택, 자동 확장 기능을 이번 범위에서 제외해야 한다.
- **FR-012**: 시스템은 이번 기능 범위에서 새로운 테스트 체계 도입을 필수 산출물로 요구하지 않아야 한다.
- **FR-013**: 시스템은 문서에 배포 전 필요한 외부 준비사항, 운영자가 입력해야 하는 값, 실행 순서를 명확히 구분해 제공해야 한다.
- **FR-014**: 시스템은 코드 검토 단계와 운영 배포 단계를 분리해야 하며, PR 생성 시점에는 운영 배포를 실행하지 않아야 한다.
- **FR-015**: 시스템은 main 병합 후 자동 배포를 기본 운영 흐름으로 정의해야 한다.
- **FR-016**: 시스템은 초기 필수 검증 항목으로 인프라 포맷 및 기본 검증, 애플리케이션 빌드, 컨테이너 이미지 빌드, 배포 명세 기본 검사를 포함해야 한다.
- **FR-017**: 시스템은 컨테이너 이미지 저장소 선택 시 비용, 운영 단순성, 접근 제어, 배포 연동 편의성을 비교할 수 있어야 한다.

### Constitution Alignment *(mandatory)*

- **CA-001**: 이번 기능은 기존 `/frontend`와 `/backend` 코드 경계를 유지하며, 배포 대상으로 두 애플리케이션을 그대로 다룬다. 새 기능이 프론트엔드에서 백엔드 소스를 직접 참조하게 만들어서는 안 된다.
- **CA-002**: 이번 기능은 기존 애플리케이션 API 계약을 변경하는 것이 목적이 아니므로 신규 백엔드 엔드포인트 추가는 기본 범위에 없다. 배포 후에도 백엔드 JSON 응답 형식 `{ success: boolean, data: any, error: string | null }`은 유지되어야 한다.
- **CA-003**: 배포 자동화와 운영에 Terraform, Kubernetes 관련 도구, 배포 관리 도구 구성이 필요할 수 있다. 저장소 의존성 또는 로컬 설치가 추가로 필요해지면 반드시 `NEEDS USER APPROVAL`로 다루고 자동 설치를 가정하지 않는다.
- **CA-003a**: CI/CD, 클러스터, 이미지 저장소 연동 설계가 필요하더라도 저장소 의존성 추가나 로컬 도구 설치가 필요해지는 시점에는 먼저 사용자 승인을 받아야 한다.
- **CA-004**: 이번 기능은 런타임 배포와 운영 문서가 중심이며, 애플리케이션 UX 자체를 바꾸지 않는다. 따라서 기존 인라인 편집 중심 UX를 변경하지 않는다.
- **CA-005**: 배포 대상 애플리케이션의 저장 책임은 계속 백엔드가 가진다. 이번 기능은 배포 환경과 운영 절차를 정의하는 것이며, 저장소 구조를 복잡한 신규 데이터 계층으로 바꾸지 않는다.

### Key Entities *(include if feature involves data)*

- **Deployment Environment**: AWS 상에 구성되는 단일 운영 환경으로, 네트워크 접근 경로, 서버, 클러스터, 애플리케이션 배포 상태를 포함한다.
- **Bootstrap Configuration**: 서버 초기화와 클러스터 준비에 필요한 입력값과 실행 순서를 설명하는 구성 정보다.
- **Application Release Definition**: 애플리케이션 이미지, 설정값, 배포 대상 리소스, 외부 노출 방식 등을 정의하는 배포 단위다.
- **Delivery Workflow**: 검토용 검증 흐름과 main 병합 후 자동 배포 흐름을 구분해 설명하는 릴리스 절차다.
- **Secret Inventory**: 애플리케이션 실행에 필요한 민감 정보 목록과 주입 방법, 갱신 방법을 정리한 운영 대상이다.
- **Image Registry Option**: 컨테이너 이미지 저장 위치 후보와 비용, 접근 방식, 배포 연동 특성을 비교하기 위한 운영 결정 대상이다.
- **Operations Runbook**: 초기 배포, 재배포, 점검, 기본 복구 절차를 설명하는 운영 문서 세트다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 빈 AWS 준비 상태에서 운영 문서만으로 단일 운영 환경 생성부터 외부 접속 확인까지 60분 이내에 완료할 수 있다.
- **SC-002**: 초기 배포 후 외부 사용자는 공개된 접속 경로로 애플리케이션 첫 화면에 접근할 수 있다.
- **SC-003**: 동일한 배포 구성을 다시 적용해도 수동 서버 수정 없이 애플리케이션 재배포를 15분 이내에 수행할 수 있다.
- **SC-003a**: PR이 열렸을 때 운영 배포 없이 필수 검증 결과를 확인할 수 있어야 한다.
- **SC-003b**: main 병합 후 자동 배포 흐름은 별도 수동 서버 접속 없이 시작될 수 있어야 한다.
- **SC-004**: 운영 문서를 처음 보는 팀원이 필수 입력값, 배포 순서, 기본 점검 순서를 빠뜨리지 않고 따라갈 수 있다.
- **SC-005**: 이번 MVP 문서만으로 어떤 항목이 비용 절감을 위한 의도적 제한인지 팀이 명확히 이해할 수 있다.
- **SC-006**: 팀은 문서만 보고 도메인·HTTPS 적용 여부와 이미지 저장소 후보를 비용과 단순성 기준으로 비교할 수 있어야 한다.

## Assumptions

- AWS 계정과 기본 결제 설정은 사용 가능하지만, 네트워크와 서버 리소스는 새로 생성해야 한다.
- 첫 릴리스에서는 단일 사용자 또는 소규모 내부 검증 수준의 트래픽을 가정한다.
- 공개 도메인이 없어도 초기 검증은 공인 IP 또는 임시 접근 경로로 수행할 수 있다.
- 초기 외부 접속은 하나의 공인 엔드포인트만 제공하면 충분하다.
- 도메인과 HTTPS는 초기 필수조건이 아니며, 후속 우선순위 판단 대상이다.
- 애플리케이션 빌드 산출물 또는 컨테이너 이미지 준비 방식은 이후 계획 단계에서 구체화하되, 이번 spec에서는 배포 가능 상태 확보를 우선한다.
- 코드 검토용 검증과 운영 배포용 자동화는 분리된 흐름으로 설계한다.
- AWS 연동 인증은 장기 정적 자격증명보다 단기 신뢰 기반 접근을 우선한다.
- 본격적인 테스트 코드, 멀티 환경 분리, 고급 모니터링 도입은 후속 단계로 미룬다.
- 주요 개발 및 검증 흐름은 WSL Ubuntu 기반 명령 환경을 전제로 한다.
- 이번 feature는 사용자 지정 제약에 따라 선언적 인프라 관리, 단일 EC2 기반 클러스터, 동일 서버 내 선언적 배포 관리 구성을 전제로 한다.
