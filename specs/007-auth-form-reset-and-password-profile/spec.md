## User Scenarios & Testing *(mandatory)*

### User Story 1 - 인증 탭 전환 시 폼 상태 초기화 (Priority: P1)

사용자는 로그인과 회원가입 탭을 오갈 때 이전 탭의 이메일/비밀번호/오류가 다음 탭에 그대로 남지 않는 깔끔한 폼 상태를 볼 수 있다.

**Why this priority**: 현재 UX 버그를 바로 해소하는 가장 직접적인 사용자 가치다.

**Independent Test**: 로그인 탭에서 이메일과 비밀번호를 입력한 뒤 회원가입 탭으로 이동하면 기존 입력과 오류가 초기화되고, 반대로 회원가입 탭에서 입력 후 로그인 탭으로 돌아와도 값이 남지 않으면 된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 로그인 탭에 이메일과 비밀번호를 입력했을 때, **When** 회원가입 탭으로 전환하면, **Then** 회원가입 폼은 빈 상태로 시작한다.
2. **Given** 사용자가 회원가입 탭에서 이름, 표시 이름, 이메일, 비밀번호, 비밀번호 확인을 입력했을 때, **When** 로그인 탭으로 전환하면, **Then** 로그인 폼에는 이전 회원가입 값이 남지 않는다.
3. **Given** 한 탭에서 로컬 검증 오류가 표시된 상태일 때, **When** 다른 탭으로 전환하면, **Then** 이전 탭의 오류 메시지가 함께 남지 않는다.

---

### User Story 2 - 프로필 화면에서 비밀번호 변경 (Priority: P2)

로그인한 사용자는 프로필 화면 안에서 현재 비밀번호를 확인한 뒤 새 비밀번호로 변경할 수 있다.

**Why this priority**: 계정 관리 기능으로서 프로필 화면의 실질적 완성도를 높인다.

**Independent Test**: 프로필 화면의 비밀번호 변경 섹션에서 현재 비밀번호, 새 비밀번호, 새 비밀번호 확인을 입력해 저장하고, 이후 새 비밀번호로 로그인 가능하면 된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 올바른 현재 비밀번호와 유효한 새 비밀번호를 입력했을 때, **When** 비밀번호 변경을 저장하면, **Then** 비밀번호가 갱신되고 성공 메시지가 표시된다.
2. **Given** 현재 비밀번호가 틀렸을 때, **When** 비밀번호 변경을 시도하면, **Then** 오류 메시지가 표시되고 기존 비밀번호는 유지된다.
3. **Given** 새 비밀번호와 새 비밀번호 확인이 일치하지 않을 때, **When** 제출하면, **Then** 서버 요청 전에 오류가 표시된다.

### User Story 3 - README 스펙/기능 설명 보강 (Priority: P3)

개발자는 README만 읽어도 `001`부터 `006`까지 어떤 스펙이 있었고 실제로 어떤 기능이 구현됐는지 빠르게 파악할 수 있다.

**Why this priority**: 현재 저장소 이해 비용을 줄이고 이후 작업 컨텍스트 확보를 쉽게 만든다.

**Independent Test**: README에 각 스펙 요약과 현재 구현 상태가 정리되어 있어, 별도 스펙 파일을 열지 않아도 전체 흐름을 이해할 수 있으면 된다.

**Acceptance Scenarios**:

1. **Given** 저장소 루트의 README를 열었을 때, **When** 구현 범위와 스펙 목록을 읽으면, **Then** 각 스펙의 목적과 현재 반영 여부를 확인할 수 있다.
2. **Given** 새 작업자가 프로젝트를 파악하려 할 때, **When** README를 읽으면, **Then** 인증, 노트, 이미지, 프로필 기능 상태를 빠르게 이해할 수 있다.

---

### Edge Cases

- 탭 전환 직후에도 브라우저 자동완성 값과 컴포넌트 state가 어긋나지 않아야 한다.
- 비밀번호 변경 실패 시 이름/표시 이름 편집 상태는 영향을 받지 않아야 한다.
- 비밀번호 변경 성공 후에는 민감한 입력값이 즉시 초기화되어야 한다.
- README는 구현되지 않은 기능을 이미 반영된 것처럼 과장하면 안 된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 로그인/회원가입 탭 전환 시 현재 탭의 입력값과 로컬 오류 상태를 초기화해야 한다.
- **FR-002**: 시스템은 로그인 탭과 회원가입 탭이 서로의 입력 상태를 공유하지 않도록 해야 한다.
- **FR-003**: 시스템은 프로필 화면에 비밀번호 변경 전용 섹션을 제공해야 한다.
- **FR-004**: 비밀번호 변경 시 `currentPassword`, `newPassword`, `newPasswordConfirm` 입력을 받아야 한다.
- **FR-005**: 시스템은 현재 비밀번호가 맞는 경우에만 비밀번호 변경을 허용해야 한다.
- **FR-006**: 시스템은 새 비밀번호와 새 비밀번호 확인이 일치할 때만 비밀번호 변경 요청을 전송해야 한다.
- **FR-007**: 시스템은 비밀번호 변경 성공 후 민감 입력 필드를 초기화해야 한다.
- **FR-008**: README는 `001`~`006` 스펙의 목적과 현재 구현 범위를 요약해야 한다.

### Constitution Alignment *(mandatory)*

- **CA-001**: 비밀번호 검증/변경 로직은 `/backend`에서 처리하고 프론트엔드는 인증 API를 통해서만 요청한다.
- **CA-002**: 변경되는 백엔드 엔드포인트는 `PATCH /api/auth/me/password` 또는 동등한 인증 API이며 모든 응답은 `{ success, data, error }` envelope을 유지한다.
- **CA-003**: 신규 의존성은 추가하지 않는다. 기존 bcryptjs, jsonwebtoken, React만 사용한다.
- **CA-004**: UX는 기존 인증 폼/프로필 화면을 확장하는 방식으로 처리하고, 모달이나 별도 라우터는 추가하지 않는다.
- **CA-005**: 사용자 비밀번호 및 프로필 저장 책임은 계속 백엔드 `users.json` 저장소에 있으며, 저장소 접근은 repository/service 레이어로 제한한다.

### Key Entities *(include if feature involves data)*

- **AuthFormStateBuckets**: 로그인 폼 상태와 회원가입 폼 상태를 분리한 입력/오류 묶음
- **PasswordChangeInput**: 비밀번호 변경 입력. `currentPassword`, `newPassword`, `newPasswordConfirm`

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 탭 전환 후 이전 탭 입력값이 다음 탭에 남는 사례가 0건이어야 한다.
- **SC-002**: 올바른 현재 비밀번호 입력 시 비밀번호 변경이 한 번의 시도로 완료된다.
- **SC-003**: 새 비밀번호/확인 불일치 케이스는 서버 요청 전에 100% 차단된다.
- **SC-004**: README만 읽고도 현재 구현된 스펙 범위를 파악할 수 있어야 한다.

## Assumptions

- 비밀번호 변경은 현재 로그인 세션 안에서만 제공한다.
- 새 비밀번호 최소 길이는 기존과 동일하게 6자 이상을 유지한다.
- 비밀번호 변경 성공 후 토큰은 유지하고 세션을 강제 종료하지 않는다.
- README 보강은 이미 존재하는 스펙 `001`~`006` 기준으로만 수행한다.
