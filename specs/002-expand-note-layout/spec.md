# Feature Specification: 노트 편집 레이아웃 확장

**Feature Branch**: `002-expand-note-layout`
**Created**: 2026-04-14
**Status**: Clarified
**Input**: User description: "study-note 사용성 개선 - 노트 작성 시 화면이 좁아 넓히는 옵션이나 레이아웃 추가 필요"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 넓은 레이아웃으로 전환 (Priority: P1)

사용자가 노트를 작성하는 중에 현재 편집 영역이 좁다고 느낄 때, 버튼 클릭 한 번으로 넓은 레이아웃으로 전환할 수 있다.

**Why this priority**: 핵심 불편함(좁은 화면)을 직접 해결하는 기능으로, 다른 레이아웃 옵션의 전제가 된다.

**Independent Test**: 편집 화면에서 레이아웃 전환 버튼을 클릭하면 편집 영역 너비가 넓어지는 것을 시각적으로 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 기본(표준) 레이아웃으로 노트 편집 화면을 열었을 때, **When** 레이아웃 전환 버튼을 클릭하면, **Then** NoteComposer가 전체 너비를 차지하고 NoteList가 그 아래에 배치되는 단일 열 레이아웃으로 전환된다.
2. **Given** 사용자가 넓은 레이아웃 상태일 때, **When** 동일한 버튼을 다시 클릭하면, **Then** 기본 레이아웃으로 돌아온다.
3. **Given** 사용자가 레이아웃을 전환할 때, **When** 전환이 완료되면, **Then** 작성 중이던 노트 내용이 그대로 유지된다.

---

### User Story 2 - 레이아웃 선택 기억 (Priority: P2)

사용자가 선택한 레이아웃이 다음 방문 시에도 유지되어 매번 설정을 반복하지 않아도 된다.

**Why this priority**: 반복적인 설정 변경 없이 개인 선호에 맞는 환경을 제공해 생산성을 높인다.

**Independent Test**: 넓은 레이아웃으로 전환 후 페이지를 새로고침하거나 다시 접속해도 넓은 레이아웃이 유지되는 것을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 넓은 레이아웃을 선택한 후, **When** 브라우저를 닫고 다시 접속하면, **Then** 이전에 선택한 넓은 레이아웃이 적용된 상태로 편집 화면이 열린다.
2. **Given** 사용자가 기본 레이아웃으로 되돌린 후, **When** 페이지를 새로고침하면, **Then** 기본 레이아웃이 유지된다.

---

### User Story 3 - 3단계 레이아웃 선택 (Priority: P3)

사용자가 좁음 / 기본 / 넓음 세 가지 레이아웃 중 원하는 것을 선택할 수 있다.

**Why this priority**: 다양한 화면 크기와 작업 스타일에 맞는 세분화된 선택지를 제공한다. P1/P2로 기본 가치를 먼저 제공한 후 확장한다.

**Independent Test**: 레이아웃 선택 UI에서 세 가지 옵션을 각각 선택했을 때 편집 영역 너비가 각각 다르게 표시되는 것을 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 레이아웃 선택 UI를 열었을 때, **When** 세 가지 옵션(좁음/기본/넓음)이 표시되면, **Then** 현재 선택된 옵션이 시각적으로 강조 표시된다.
2. **Given** 사용자가 '넓음' 옵션을 선택했을 때, **When** 선택이 완료되면, **Then** 편집 영역이 뷰포트의 80% 이상 너비로 표시된다.
3. **Given** 사용자가 '좁음' 옵션을 선택했을 때, **When** 선택이 완료되면, **Then** 편집 영역이 뷰포트의 50% 이하 너비로 표시된다.

---

### Edge Cases

- 모바일/좁은 뷰포트에서 넓은 레이아웃을 선택하면 어떻게 동작하는가? (가정: 최소 너비 이하에서는 레이아웃 선택 무효화 또는 단일 모드 적용)
- 레이아웃 전환 중 자동 저장이 발생하면 내용이 유실되지 않는가?
- 이미 열려 있는 여러 탭에서 레이아웃을 변경하면 다른 탭에 영향을 주는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 노트 편집 화면에서 사용자가 레이아웃 모드를 전환할 수 있는 컨트롤을 제공해야 한다.
- **FR-002**: 시스템은 최소 2가지(기본, 넓음) 레이아웃 모드를 지원해야 한다.
- **FR-003**: 레이아웃 전환은 노트 내용 손실 없이 즉시 적용되어야 한다.
- **FR-004**: 시스템은 사용자가 마지막으로 선택한 레이아웃 모드를 다음 접속 시에도 유지해야 한다.
- **FR-005**: 넓은 레이아웃 모드에서 편집 영역은 화면 너비의 80% 이상을 차지해야 한다.
- **FR-006**: 레이아웃 전환 컨트롤은 편집 화면 내에서 키보드로도 접근 가능해야 한다.

### Constitution Alignment *(mandatory)*

- **CA-001**: 레이아웃 설정은 프론트엔드 UI 상태이며 `/frontend`에서만 관리된다. 설정 저장이 필요한 경우 `/backend`의 HTTP API를 통해서만 접근한다.
- **CA-002**: 레이아웃 선호는 브라우저 localStorage에 저장되며 백엔드 API 호출이 발생하지 않는다. 새로운 백엔드 엔드포인트 추가 불필요.
- **CA-003**: 레이아웃 전환에 새로운 외부 의존성은 불필요하다. CSS 레이아웃 속성과 기존 상태 관리 방식으로 구현 가능하다.
- **CA-004**: 레이아웃 전환은 편집 화면 내 인라인 토글/버튼으로 제공된다. 별도 모달 불필요.
- **CA-005**: 레이아웃 선호는 브라우저 localStorage의 단순 키-값으로 저장한다. 백엔드 스토리지 변경 없음.

### Key Entities *(include if feature involves data)*

- **LayoutPreference**: 사용자가 선택한 레이아웃 모드 (`narrow` | `default` | `wide`), 마지막 변경 시각
- **NoteEditorLayout**: 레이아웃 모드별 화면 너비 비율 정의

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 3번 이하의 클릭으로 원하는 레이아웃으로 전환할 수 있다.
- **SC-002**: 넓은 레이아웃 모드에서 편집 영역이 화면 너비의 80% 이상을 차지한다.
- **SC-003**: 레이아웃 전환 후 페이지 재방문 시 선택한 레이아웃이 100% 유지된다.
- **SC-004**: 레이아웃 전환 시 노트 내용 손실이 0건이다.
- **SC-005**: 레이아웃 전환이 체감 지연 없이 즉시(1초 이내) 완료된다.

## Clarifications

### Session 2026-04-14

- Q: 넓은 레이아웃 모드에서 어떤 변화가 있어야 하는가? → A: NoteComposer가 전체 너비를 차지하고, NoteList는 아래에 쌓이는 단일 열 레이아웃으로 전환
- Q: 레이아웃 선호를 어디에 저장할 것인가? → A: 브라우저 localStorage (프론트엔드 전용, 백엔드 변경 없음)

## Assumptions

- 현재 노트 편집 화면은 고정 너비 중앙 정렬 레이아웃을 사용하고 있다.
- 주 사용 환경은 데스크톱/노트북이며, 모바일은 MVP 범위 밖이다.
- 레이아웃 선호는 전체 노트에 전역으로 적용된다(노트별 개별 설정 아님).
- 레이아웃 선호는 브라우저 localStorage에 저장한다. 백엔드 변경 없음 (기기 간 동기화 미지원).
- 레이아웃 옵션은 2가지(기본, 넓음)로 시작하고, P3 스토리에서 3가지로 확장한다.
