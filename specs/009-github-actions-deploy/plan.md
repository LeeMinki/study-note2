# Implementation Plan: GitHub Actions Automatic Deployment

**Branch**: `009-github-actions-deploy` | **Date**: 2026-04-16 | **Spec**: [spec.md](/home/hyerin/speckit/study-note2/specs/009-github-actions-deploy/spec.md)

## Summary

008에서 준비한 단일 AWS EC2, k3s, Argo CD, ECR 기반 MVP 배포 구조 위에 GitHub Actions 자동배포 운영 흐름을 정식화한다. PR에서는 production 배포 없이 build 중심 검증만 수행하고, main 병합 후에만 ECR 이미지 publish와 GitOps 상태 갱신을 수행한다. AWS 인증은 GitHub OIDC 기반 deploy role을 사용하고, Argo CD는 저장소의 GitOps path를 동기화하는 단순 구조만 유지한다. 이번 계획은 workflow 역할 분리, required status checks, OIDC/ECR/Argo CD 연계, 실패 확인 포인트, 010 테스트 확장 지점을 문서와 계약으로 고정한다.

## Technical Context

**Language/Version**: YAML for GitHub Actions and Kubernetes manifests, HCL for existing Terraform IAM/OIDC structure, shell for WSL/Linux verification commands, existing Node.js 22 frontend/backend build runtime
**Primary Dependencies**: GitHub Actions hosted runners, Docker build on GitHub Actions, AWS OIDC deploy role from 008, Amazon ECR repositories `study-note-backend` and `study-note-frontend`, Argo CD core, k3s, existing frontend/backend npm dependencies
**Storage**: No application storage change; deployment state is represented by GitOps manifest updates under `infra/kubernetes/study-note/overlays/mvp`
**Testing**: No new test framework in 009; PR checks start with dependency install, build/startup sanity, container image build, Terraform fmt/validate, Kubernetes manifest render sanity
**Target Platform**: GitHub Actions for CI/CD, AWS ap-northeast-2 MVP environment from 008, single EC2 k3s cluster with Argo CD core
**Project Type**: Web monorepo with deployment automation and GitOps manifests
**Performance Goals**: PR checks complete within 10 minutes; main deployment workflow starts within 15 minutes of merge and records success/failure in GitHub
**Constraints**: No production deploy from PR; no long-lived AWS access key in GitHub secrets; lowest practical workflow count; no multi-environment split; no advanced rollback automation; no large observability stack
**Scale/Scope**: One repository, one main branch, one MVP deployment environment, two application images, three required PR status checks, one main deployment workflow

## Phase 0 Research Output

Research decisions are recorded in [research.md](/home/hyerin/speckit/study-note2/specs/009-github-actions-deploy/research.md). Key outcomes:

- Use two workflow files: `PR Checks` for pull requests and `Deploy Main` for pushes to main.
- Use GitHub OIDC with the 008 deploy role and repository-scoped trust policy.
- Use Amazon ECR repositories `study-note-backend` and `study-note-frontend`.
- Use GitOps manifest update as the Argo CD sync trigger; do not call Argo CD API or expose Argo CD UI in 009.
- Keep required status checks stable and branch-protection friendly: `Terraform fmt and validate`, `App and image build`, `Kubernetes manifest sanity`.
- Defer real unit/integration/e2e tests to 010 while leaving workflow insertion points.

## Phase 1 Design Output

### Workflow File 분리 전략

- `pr-checks.yml` is the only PR workflow.
- `deploy-main.yml` is the only main deployment workflow.
- PR workflow must never request AWS write permissions, publish images, mutate GitOps state, or touch the production cluster.
- Main workflow performs AWS OIDC authentication, ECR login, image build/push, GitOps image tag update, manifest render validation, and commit with `[skip deploy]` to avoid recursion.
- Main workflow must fail fast when required GitHub variables are missing. A skipped deployment is allowed only for the intentional `[skip deploy]` recursion guard.

### PR 검증 Workflow 역할

Required status checks:

| Check name | Purpose | Production impact |
|------------|---------|-------------------|
| `Terraform fmt and validate` | Terraform formatting, provider init without backend, validation | None |
| `App and image build` | frontend build, backend startup sanity, frontend/backend Docker image build | None |
| `Kubernetes manifest sanity` | render MVP overlay and ensure manifest output exists | None |

009 does not add new lint packages. For this MVP, the "static or equivalent checks" requirement is satisfied by existing no-new-dependency checks: Terraform formatting/validation, frontend production build, backend startup sanity, local Docker image builds, and Kubernetes manifest render sanity. If repository-native lint scripts are added later, 010 may insert them before build steps.

010 test insertion points:

- Add frontend unit/component tests after frontend dependency install and before frontend build.
- Add backend unit/integration tests after backend dependency install and before startup sanity.
- Add API contract tests after backend startup sanity.
- Add deployment dry-run or schema validation after manifest render if a lightweight validator is approved.

### Main 배포 Workflow 역할

Main merge deployment sequence:

1. Checkout repository with write token for GitOps tag commit.
2. Configure AWS credentials through GitHub OIDC and `AWS_DEPLOY_ROLE_ARN`.
3. Login to Amazon ECR.
4. Ensure or use ECR repositories `study-note-backend`, `study-note-frontend`.
5. Build backend and frontend images tagged with `${{ github.sha }}`.
6. Push both images to ECR.
7. Update `infra/kubernetes/study-note/overlays/mvp/kustomization.yaml` image references.
8. Render manifests to verify updated GitOps state.
9. Commit updated image tags with `[skip deploy]`.
10. Let Argo CD core reconcile the GitOps path from main.

ECR repository ownership is pragmatic for the MVP: Terraform defines IAM permissions and the intended repository names, while the main deployment workflow may create the two repositories if they are missing. If stricter IaC ownership is required later, repository resources should move into Terraform in a follow-up spec.

Branch protection must account for the GitOps update commit. The `github-actions[bot]` push to `infra/kubernetes/study-note/overlays/mvp/kustomization.yaml` either needs to be allowed by repository rules, or operators must use the documented manual recovery path when branch protection blocks the push.

### OIDC용 AWS IAM Role 및 Trust Policy 개요

Use the 008 identity module baseline:

- OIDC provider: `https://token.actions.githubusercontent.com`
- Audience: `sts.amazonaws.com`
- Subject condition: `repo:LeeMinki/study-note2:ref:refs/heads/main`
- GitHub role permissions:
  - ECR create/describe/read/write actions needed for image publish
  - read-only EC2 describe permissions for deployment diagnostics
- No long-lived AWS access key is stored in GitHub secrets.
- GitHub repository variables:
  - `AWS_REGION`
  - `AWS_DEPLOY_ROLE_ARN`

### ECR Login / Build / Push 흐름

- Main workflow authenticates to AWS via OIDC.
- ECR login obtains short-lived registry credentials.
- Images are built from existing `backend/Dockerfile` and `frontend/Dockerfile`.
- The image tag is the Git commit SHA for traceability.
- Repository names remain compatible with 008:
  - `study-note-backend`
  - `study-note-frontend`
- PR workflow may build images locally with temporary tags but must not push them.

### Argo CD Sync 방식

- Argo CD Application remains `infra/kubernetes/argocd/applications/study-note-mvp.yaml`.
- The Application watches `infra/kubernetes/study-note/overlays/mvp`.
- The main workflow updates image references in the GitOps overlay.
- Argo CD sync is triggered by repository state change, not by direct Argo CD API calls.
- 009 excludes advanced Argo CD rollout strategies, app-of-apps, manual approval gates, notification integrations, and UI exposure.

### 실패 시 확인 포인트

- PR check failure:
  - Check failed job name first.
  - For build failures, inspect frontend/backend dependency install and build logs.
  - For manifest failures, inspect rendered kustomize output.
- OIDC/auth failure:
  - Confirm `AWS_REGION` and `AWS_DEPLOY_ROLE_ARN` repository variables.
  - Confirm IAM trust subject matches main branch.
  - Confirm workflow permissions include `id-token: write`.
- ECR failure:
  - Confirm repository names and ECR permissions.
  - Confirm ECR login step succeeded before push.
- GitOps update failure:
  - Confirm `kustomization.yaml` image block was updated.
  - Confirm `[skip deploy]` commit is produced only when content changes.
- Argo CD sync/runtime failure:
  - Confirm Argo CD Application exists.
  - Confirm k3s namespace, ECR pull secret, and image tags exist.
  - Confirm pod image pull and readiness status on the EC2 k3s cluster.
