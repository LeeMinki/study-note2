# Feature Specification: 도메인 기반 접속 및 HTTPS 적용

**Feature Branch**: `011-domain-https`
**Created**: 2026-04-17
**Status**: Clarified
**Input**: Study Note의 현재 IP 기반 접속을 도메인 기반 접속으로 바꾸고 HTTPS를 적용하고 싶다.

## Clarifications

### Session 2026-04-17

- **단일 도메인 우선**: 이번 범위는 단일 루트 도메인 하나만 지원한다. 서브도메인 분리와 멀티 도메인은 후속 spec에서 다룬다.
- **www 포함 여부**: 루트 도메인(`domain.com`)을 primary로 하고, `www.domain.com`은 루트 도메인으로 리디렉션한다. DNS A 레코드 1개(루트) + CNAME 1개(www → 루트)로 구조를 단순하게 유지한다. TLS 인증서는 루트와 www를 모두 커버한다.
- **인증서 자동 갱신**: 수동 갱신 없이 운영할 수 있는 표준 자동 갱신 방식을 우선 검토한다. 인증서 발급 비용 없이 자동 갱신이 가능한 방식을 선택한다.
- **기존 구조 유지**: AWS 단일 EC2 + k3s + Argo CD 배포 구조를 변경하지 않는다. 인프라 추가는 기존 Kubernetes 워크로드 내에서 최소화한다.
- **비용 최소화**: 인증서는 무료 발급 방식을 사용한다. DNS 관리 비용은 월 $1 이하를 목표로 한다. 추가 로드밸런서, CDN, 유료 인증서는 범위에서 제외한다.
- **DNS 및 ingress 단순화**: DNS 레코드는 A 레코드(루트) + CNAME(www) 2개로 유지한다. ingress 규칙은 단일 서비스 경로로 단순하게 유지하며 복잡한 라우팅 규칙을 추가하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 도메인으로 HTTPS 접속 (Priority: P1)

사용자가 IP 주소 대신 도메인 이름을 통해 Study Note에 접속하면, 브라우저 보안 경고 없이 암호화된 연결로 서비스를 이용할 수 있어야 한다.

**Why this priority**: 도메인 접속과 HTTPS는 서비스의 기본 신뢰성 요건이다. IP 주소는 재배포 시 바뀔 수 있고, 브라우저 보안 경고는 사용자 경험을 심각하게 해친다.

**Independent Test**: 사용자가 도메인 주소를 브라우저 주소창에 입력하면 브라우저 잠금 아이콘이 표시된 HTTPS 연결로 Study Note가 정상 로드되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 도메인 설정이 완료된 상태에서, **When** 사용자가 브라우저에 도메인 주소를 입력하면, **Then** 브라우저 보안 경고 없이 Study Note 앱이 로드된다.
2. **Given** 도메인 설정이 완료된 상태에서, **When** 사용자가 해당 URL에 접속하면, **Then** 브라우저 주소창에 자물쇠 아이콘이 표시된다.
3. **Given** 유효한 HTTPS 인증서가 적용된 상태에서, **When** 외부 도구로 HTTPS 연결을 검증하면, **Then** 신뢰할 수 있는 기관에서 발급된 유효한 인증서임이 확인된다.

---

### User Story 2 - HTTP 접속 시 HTTPS 자동 전환 (Priority: P2)

사용자가 HTTP로 도메인에 접속해도 자동으로 HTTPS로 전환되어 보안 연결이 유지된다.

**Why this priority**: 사용자가 실수로 HTTP를 입력하거나 이전 북마크를 사용하는 경우에도 보안 연결이 보장되어야 한다.

**Independent Test**: 브라우저에서 `http://도메인`으로 직접 접속했을 때 자동으로 `https://도메인`으로 리디렉션되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** HTTPS가 설정된 상태에서, **When** 사용자가 HTTP로 접속하면, **Then** 자동으로 HTTPS로 리디렉션된다.
2. **Given** HTTP→HTTPS 리디렉션이 작동하는 상태에서, **When** 리디렉션이 완료되면, **Then** 사용자는 보안 경고 없이 정상 페이지에 도달한다.

---

### User Story 3 - 인증서 자동 갱신 (Priority: P3)

운영자가 별도의 수동 조작 없이도 HTTPS 인증서가 만료 전에 자동으로 갱신되어 서비스 가용성이 유지된다.

**Why this priority**: 인증서 만료는 서비스 전체에 보안 경고 또는 접속 불가 상태를 초래한다. 갱신 자동화는 운영 부담을 낮추고 가용성을 높인다.

**Independent Test**: 인증서 유효 기간을 확인하고 자동 갱신 설정이 활성화되어 있는지 검증한다. 갱신 주기 도래 시 인증서가 새로운 유효 기간으로 교체되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 자동 갱신 설정이 완료된 상태에서, **When** 인증서 만료 기간이 가까워지면, **Then** 운영자 개입 없이 인증서가 갱신된다.
2. **Given** 인증서가 갱신된 이후, **When** 사용자가 접속하면, **Then** 이전과 동일하게 브라우저 경고 없이 HTTPS 연결이 유지된다.
3. **Given** 인증서 갱신에 실패한 경우, **When** 운영자가 운영 문서를 확인하면, **Then** 실패 원인과 수동 복구 절차를 확인할 수 있다.

---

### Edge Cases

- DNS 전파가 완료되기 전 도메인 접속 시 연결 실패가 발생할 수 있다 — 운영 문서에 전파 확인 방법이 포함되어야 한다.
- EC2 인스턴스 IP 주소가 변경된 경우 DNS A 레코드 갱신이 필요하다 — 운영 문서에 이 절차가 포함되어야 한다.
- 인증서 갱신 실패 시 서비스 접속이 차단될 수 있다 — 실패 감지 방법과 수동 복구 절차가 운영 문서에 있어야 한다.
- 도메인 없이 IP로 직접 접속하는 경우 별도 처리 없이 기존 HTTP 접속이 가능한 상태가 유지된다 — 이 케이스는 보안 강화 대상으로 planning 단계에서 처리 방침을 결정한다.
- 도메인 등록 갱신을 잊어 도메인이 만료되는 경우 서비스 전체가 접속 불가가 된다 — 도메인 등록 만료 관리는 운영자 책임으로 가정하되 운영 문서에 주의사항을 명시한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자가 등록된 도메인 이름으로 Study Note에 접속할 수 있어야 한다.
- **FR-002**: 시스템은 도메인 접속 시 HTTPS 암호화 연결을 제공해야 한다.
- **FR-003**: 시스템은 HTTP로 접속하는 사용자를 HTTPS로 자동 리디렉션해야 한다.
- **FR-004**: 시스템은 신뢰할 수 있는 기관에서 발급된 HTTPS 인증서를 사용해야 한다.
- **FR-005**: 시스템은 HTTPS 인증서를 만료 전에 자동으로 갱신해야 한다.
- **FR-006**: 시스템은 인증서 갱신 실패 시 운영자가 확인할 수 있는 실패 상태 정보를 제공해야 한다.
- **FR-007**: 시스템은 도메인 설정, DNS 관리, 인증서 갱신 절차를 담은 운영 문서를 제공해야 한다.
- **FR-008**: 시스템은 단일 루트 도메인을 primary로 동작해야 하며 멀티 도메인 운영은 이번 범위에 포함하지 않는다.
- **FR-009**: 시스템은 기존 AWS 단일 EC2 + k3s + Argo CD 배포 구조를 유지해야 하며 인프라 추가는 기존 Kubernetes 환경 내에서 최소화해야 한다.
- **FR-010**: 시스템은 인증서 발급에 추가 비용이 발생하지 않는 방식을 사용해야 하며 DNS 관리 비용은 월 $1 이하를 목표로 해야 한다.
- **FR-011**: DNS 관리는 운영자가 기존 AWS 환경 내에서 일관되게 관리할 수 있는 방식을 따라야 한다.
- **FR-012**: `www.도메인` 접속은 루트 도메인(`도메인`)으로 리디렉션되어야 하며, TLS 인증서는 루트 도메인과 www 서브도메인 양쪽을 커버해야 한다.
- **FR-013**: DNS 레코드는 루트 도메인용 A 레코드와 www용 CNAME 레코드 2개로 구성을 최소화해야 한다.
- **FR-014**: ingress 규칙은 단일 서비스 경로로 단순하게 유지해야 하며 이번 범위에서 복잡한 경로 라우팅을 추가하지 않는다.

### Constitution Alignment *(mandatory)*

- **CA-001**: 이 기능은 인프라 레이어(DNS, 인증서, 네트워크)에만 영향을 준다. `/frontend`와 `/backend` 소스 경계는 변경되지 않으며, 프론트엔드-백엔드 간 통신은 기존 HTTP API 방식을 유지한다.
- **CA-002**: 애플리케이션 레이어의 백엔드 엔드포인트는 변경되지 않는다. JSON 응답 형식 `{ success, data, error }` 계약은 영향받지 않는다.
- **CA-003**: 인증서 자동 갱신 도구 도입이 필요할 수 있다. 새 인프라 구성 요소는 계획 단계에서 명시하고 **사용자 승인 후 설치**한다.
- **CA-004**: 이 기능은 사용자 화면 UX에 영향을 주지 않는다. 브라우저 주소창의 잠금 아이콘이 유일한 가시적 변화다.
- **CA-005**: 애플리케이션 데이터 저장 방식은 변경되지 않는다. 백엔드 JSON 파일 스토리지는 유지된다.

### Key Entities

- **도메인**: 사용자가 Study Note에 접속하는 데 사용하는 인터넷 주소. 운영자가 등록하고 관리한다.
- **DNS 레코드**: 도메인 이름을 EC2 IP 주소에 연결하는 설정. A 레코드 기반으로 관리된다.
- **TLS 인증서**: 도메인에 대한 HTTPS 암호화를 가능하게 하는 디지털 인증서. 유효 기간이 있으며 자동 갱신 대상이다.
- **인증서 갱신 상태**: 인증서의 현재 유효 기간, 갱신 시도 결과, 다음 갱신 예정 시각 정보.
- **운영 가이드**: 도메인 설정 절차, DNS 변경 방법, 인증서 갱신 실패 시 복구 절차를 담은 문서.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 도메인 설정 완료 후 사용자는 IP 주소 없이 도메인 이름만으로 Study Note에 접근할 수 있다.
- **SC-002**: 사용자가 도메인에 접속하면 브라우저 보안 경고 없이 HTTPS 연결이 수립된다.
- **SC-003**: HTTP로 접속한 사용자는 1초 이내에 HTTPS로 자동 전환된다.
- **SC-004**: HTTPS 인증서는 만료 전에 자동 갱신이 완료되어 운영자 개입 없이 서비스가 유지된다.
- **SC-005**: EC2 IP 변경 시 운영 문서만 참고해 30분 이내에 DNS 갱신을 완료할 수 있다.
- **SC-006**: 인증서 갱신 실패 시 운영자는 운영 문서를 통해 원인 확인과 수동 복구 절차를 진행할 수 있다.

## Assumptions

- 운영자가 도메인을 이미 보유하거나 새로 등록할 수 있는 상태를 가정한다.
- DNS 관리는 기존 AWS 인프라와 통합하는 것을 기본 방향으로 가정한다. planning 단계에서 구체적인 DNS 공급자(Route 53 등)를 확정한다.
- 무료 공인 인증서를 자동 갱신 방식으로 발급할 수 있다고 가정한다.
- 기존 k3s ingress 컨트롤러가 TLS 종료를 처리할 수 있는 구조를 유지한다고 가정한다.
- 루트 도메인이 primary이며 www는 루트로 리디렉션된다. 인증서는 두 도메인을 모두 커버한다.
- DNS 레코드는 A 레코드(루트 → EC2 IP) + CNAME(www → 루트) 2개로 최소 구성한다.
- EC2 공인 IP 주소는 일반적으로 고정되어 있지만 인스턴스 재생성 시 변경될 수 있다.
- 추가 로드밸런서, CDN, 유료 인증서는 이번 범위에서 제외한다.
- 개발 및 검증 흐름은 WSL Ubuntu 기준 명령과 문서를 우선한다.
