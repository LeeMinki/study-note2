# Feature Specification: 로그인/회원가입 및 계정별 노트 분리

**Feature Branch**: `004-auth-login`
**Created**: 2026-04-15
**Status**: Clarified
**Input**: Jira SP-25 — "로그인/회원가입 기능 추가. 계정별로 보이는 노트가 달라야 하고, 회원가입도 되어야 한다. 나중엔 SSO 연동도 되어야 한다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 회원가입 (Priority: P1)

신규 사용자가 이메일과 비밀번호를 입력하여 계정을 생성하고 서비스를 이용할 수 있다.

**Why this priority**: 로그인의 전제 조건. 계정이 없으면 로그인 자체가 불가능하다.

**Independent Test**: 회원가입 폼에서 유효한 이메일과 비밀번호를 입력하면 계정이 생성되고 바로 로그인 상태로 전환된다.

**Acceptance Scenarios**:

1. **Given** 미가입 사용자가 회원가입 폼을 열었을 때, **When** 유효한 이메일과 6자 이상 비밀번호를 입력하고 제출하면, **Then** 계정이 생성되고 즉시 로그인 상태로 전환되어 메인 화면이 표시된다.
2. **Given** 이미 가입된 이메일로 회원가입을 시도할 때, **When** 제출하면, **Then** "이미 사용 중인 이메일입니다" 오류 메시지가 표시되고 계정이 중복 생성되지 않는다.
3. **Given** 잘못된 형식의 이메일이나 너무 짧은 비밀번호를 입력했을 때, **When** 제출하면, **Then** 입력 유효성 오류 메시지가 표시되고 계정이 생성되지 않는다.

---

### User Story 2 - 로그인 / 로그아웃 (Priority: P2)

기존 계정을 가진 사용자가 이메일과 비밀번호로 로그인하여 자신의 노트에 접근하고, 로그아웃으로 세션을 종료할 수 있다.

**Why this priority**: 인증의 핵심. 회원가입(P1) 후 반복 사용을 위해 필요.

**Independent Test**: 가입된 이메일과 비밀번호로 로그인하면 메인 화면이 표시되고, 로그아웃하면 로그인 화면으로 돌아온다.

**Acceptance Scenarios**:

1. **Given** 가입된 이메일과 비밀번호를 입력했을 때, **When** 로그인 버튼을 클릭하면, **Then** 인증에 성공하고 메인 노트 화면이 표시된다.
2. **Given** 잘못된 비밀번호 또는 미가입 이메일을 입력했을 때, **When** 로그인 버튼을 클릭하면, **Then** "이메일 또는 비밀번호가 올바르지 않습니다" 메시지가 표시되고 로그인이 거부된다.
3. **Given** 로그인 상태의 사용자가 로그아웃 버튼을 클릭했을 때, **When** 로그아웃이 완료되면, **Then** 로그인 화면으로 전환되고 노트 데이터에 더 이상 접근할 수 없다.
4. **Given** 페이지를 새로고침하거나 브라우저를 재시작했을 때, **When** 인증 토큰이 유효하면, **Then** 로그인 화면을 거치지 않고 바로 메인 화면이 표시된다.

---

### User Story 3 - 계정별 노트 분리 (Priority: P3)

로그인한 사용자는 자신이 작성한 노트만 조회, 수정, 삭제할 수 있다. 다른 사용자의 노트는 보이지 않는다.

**Why this priority**: 로그인(P1/P2) 이후 데이터 격리가 되어야 서비스의 핵심 가치(개인 노트)가 완성된다.

**Independent Test**: 사용자 A로 로그인하여 노트를 작성한 뒤, 사용자 B로 로그인하면 A의 노트가 보이지 않는다.

**Acceptance Scenarios**:

1. **Given** 사용자 A가 로그인하여 노트를 작성했을 때, **When** 사용자 B가 로그인하면, **Then** 사용자 B의 노트 목록에 A의 노트가 표시되지 않는다.
2. **Given** 로그인되지 않은 상태에서 노트 API에 직접 접근을 시도할 때, **When** 인증 토큰 없이 요청하면, **Then** 401 Unauthorized 응답이 반환된다.
3. **Given** 사용자 A가 사용자 B의 노트 ID로 직접 수정/삭제를 시도할 때, **When** 해당 API를 호출하면, **Then** 403 Forbidden 또는 404 응답이 반환된다.

---

### User Story 4 - SSO 연동 준비 (Priority: P4 - 향후)

향후 Google, GitHub 등의 SSO(OAuth2) 제공자를 통한 로그인이 가능하도록 인증 구조가 확장 가능해야 한다.

**Why this priority**: MVP에서는 이메일/비밀번호 인증만 구현하지만, 향후 SSO 전환 시 데이터 마이그레이션 없이 확장 가능한 구조가 필요하다.

**Independent Test**: P4는 이번 MVP 범위 밖이며, 아키텍처 설계 결정으로만 반영된다.

**Acceptance Scenarios**: MVP 범위 밖 — 아키텍처 요구사항으로만 문서화.

---

### Edge Cases

- 동시에 같은 이메일로 두 번 회원가입 요청이 들어올 경우 중복 방지는?
- 비밀번호를 잊어버린 경우? (MVP 범위 밖 — 비밀번호 재설정 미지원)
- 토큰 만료 후 자동 로그아웃 처리는?
- 기존에 생성된 노트(인증 추가 이전)는 어떻게 처리하는가? (orphaned 처리 — 특정 계정에 귀속되지 않음)
- 악의적인 다른 사용자가 URL/ID를 직접 추측하여 다른 계정의 노트에 접근 시도?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 이메일과 비밀번호를 입력받아 신규 계정을 생성하는 기능을 제공해야 한다.
- **FR-002**: 이메일은 유효한 형식이어야 하며, 비밀번호는 6자 이상이어야 한다.
- **FR-003**: 동일한 이메일로 중복 계정이 생성되어서는 안 된다.
- **FR-004**: 회원가입 성공 시 즉시 로그인 상태로 전환되어야 한다.
- **FR-005**: 시스템은 이메일과 비밀번호를 검증하여 인증 토큰을 발급하는 로그인 기능을 제공해야 한다.
- **FR-006**: 잘못된 인증 정보로 로그인 시 구체적인 계정 정보를 노출하지 않는 통합 오류 메시지를 반환해야 한다.
- **FR-007**: 로그아웃 시 클라이언트의 인증 토큰이 제거되어 더 이상 인증 API에 접근할 수 없어야 한다.
- **FR-008**: 페이지 새로고침 후에도 유효한 토큰이 있으면 로그인 상태가 유지되어야 한다.
- **FR-009**: 모든 노트 관련 API는 유효한 인증 토큰이 있는 요청만 처리해야 한다.
- **FR-010**: 노트 조회/수정/삭제는 해당 노트를 소유한 계정의 토큰으로만 가능해야 한다.
- **FR-011**: 인증 구조는 향후 SSO(OAuth2) 제공자 추가가 가능하도록 사용자 식별 방식이 추상화되어야 한다.

### Constitution Alignment *(mandatory)*

- **CA-001**: 인증 로직은 `/backend`에서만 처리한다. 비밀번호 해시, 토큰 발급/검증은 모두 서버에서 수행하며 프론트엔드는 API를 통해서만 인증한다.
- **CA-002**: 인증 엔드포인트 신규 추가: `POST /api/auth/register`, `POST /api/auth/login`. 모든 응답은 `{ success, data, error }` envelope 형식.
- **CA-003**: 신규 의존성 `bcryptjs`(비밀번호 해시), `jsonwebtoken`(JWT 발급/검증) 사용 — **사용자 승인 필요**. 프론트엔드는 신규 의존성 불필요.
- **CA-004**: 로그인/회원가입 UI는 인증되지 않은 상태에서 메인 앱 화면 대신 표시되는 인라인 폼. 별도 라우팅 라이브러리 없이 조건부 렌더링으로 처리.
- **CA-005**: 사용자 데이터는 백엔드 `users.json` 파일에 저장. 인증 토큰은 프론트엔드 localStorage에 저장. 비밀번호는 해시값만 저장하며 평문은 보존하지 않는다.

### Key Entities *(include if feature involves data)*

- **User**: id, email, passwordHash, createdAt
- **AuthToken**: JWT payload — userId, email, 만료시간 (7일)
- **Note** (수정): 기존 필드 + `userId` 추가 — 노트 소유자 식별

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 회원가입부터 메인 화면 진입까지 5초 이내에 완료된다.
- **SC-002**: 로그인부터 메인 화면 진입까지 3초 이내에 완료된다.
- **SC-003**: 사용자 A의 노트가 사용자 B의 화면에 노출되는 경우가 0건이다.
- **SC-004**: 인증 없이 노트 API에 접근 시 100% 거부된다.
- **SC-005**: 페이지 새로고침 후 유효 토큰 보유 시 로그인 화면 재표시 없이 100% 복원된다.
- **SC-006**: 비밀번호는 평문으로 저장되지 않는다 (해시 검증 가능).

## Assumptions

- 이메일/비밀번호 기반 인증만 MVP 범위. SSO(Google, GitHub)는 향후 확장.
- JWT 기반 stateless 인증. 토큰 유효 기간: 7일.
- 토큰은 프론트엔드 localStorage에 저장 (`study-note-token` 키).
- 비밀번호 해시: bcryptjs (saltRounds=10).
- 이메일 인증(이메일 발송)은 MVP 범위 밖.
- 비밀번호 재설정은 MVP 범위 밖.
- 기존 노트(userId 없음)는 인증 도입 후 orphaned 처리 — 어떤 계정에도 귀속되지 않아 노출되지 않음.
- 인증 추가 이전 데이터는 수동으로 처리해야 함 (마이그레이션 스크립트 별도 제공 가능, MVP 범위 밖).
- 비밀번호 최소 길이: 6자.
- 토큰 만료 시 자동 로그아웃 처리.
- SSO 확장을 위해 User 모델에 `provider` 필드 예약 (`"local"` | `"google"` | `"github"`).
