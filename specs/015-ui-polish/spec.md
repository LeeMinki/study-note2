## User Scenarios & Testing *(mandatory)*

### User Story 1 — 인증 화면 (Priority: P1)

사용자가 처음 앱을 열면 로그인/회원가입 화면이 나타난다. 이 화면이 앱의 첫인상을 결정하므로 가장 높은 완성도가 필요하다. 브랜드 정체성이 느껴지고, 폼 요소가 충분히 크고 읽기 쉬우며, Google SSO 버튼이 이메일 폼과 시각적으로 균형 잡혀 있어야 한다.

**Why this priority**: 모든 사용자가 반드시 거치는 화면. 첫인상이 제품 신뢰도에 직결된다.

**Independent Test**: 로그인/회원가입 화면을 열고 탭 전환, 폼 입력, Google 로그인 버튼이 모두 시각적으로 정돈되어 보이는지 확인

**Acceptance Scenarios**:

1. **Given** 미인증 상태로 앱 접근, **When** 화면 로드, **Then** 브랜드 로고/이름과 간단한 설명이 폼 위에 보이고 전체 레이아웃이 수직 중앙 정렬된다
2. **Given** 로그인 탭 활성 상태, **When** 시각 검토, **Then** 탭 전환 영역, 이메일/비밀번호 필드, Google SSO 버튼, 에러 메시지 영역이 명확한 계층 구조로 배치된다
3. **Given** 에러 발생(잘못된 비밀번호), **When** 메시지 표시, **Then** 에러 텍스트가 폼 내에서 시각적으로 구별되고 입력 필드 근처에 위치한다
4. **Given** 모바일 너비(<480px), **When** 화면 접근, **Then** 폼이 화면 너비에 맞게 패딩을 유지하며 읽기 가능하다

---

### User Story 2 — 노트 목록 및 헤더 영역 (Priority: P1)

인증 후 메인 화면의 헤더(hero) 영역과 노트 목록이 보인다. 현재 hero 제목이 영문 슬로건으로 고정되어 있고 정보 우선순위가 불명확하다. 사용자 이름 표시, 검색, 태그 필터, 로그아웃/프로필 이동이 자연스러운 흐름 안에 있어야 한다.

**Why this priority**: 로그인 후 매번 보는 핵심 화면. 정보 구조와 시각 완성도가 체감에 가장 큰 영향을 준다.

**Independent Test**: 노트가 여러 개 있는 상태에서 헤더-검색-목록 영역을 스크롤하며 각 요소의 계층이 명확한지 확인

**Acceptance Scenarios**:

1. **Given** 인증된 사용자, **When** 메인 화면 진입, **Then** 현재 로그인 계정의 표시 이름이 헤더 영역에 보이고 프로필/로그아웃 버튼이 명확히 구별된다
2. **Given** 노트가 없는 초기 상태, **When** 빈 목록 화면 확인, **Then** 빈 상태 메시지가 안내 문구와 함께 중앙에 표시되고 "첫 노트 만들기" 등 행동 유도가 시각적으로 두드러진다
3. **Given** 로딩 중, **When** 노트 목록 갱신, **Then** "노트 불러오는 중…" 텍스트와 CSS 스피너(회전 원)가 빈 화면 대신 표시된다
4. **Given** 검색 및 태그 필터 활성, **When** 결과 없음, **Then** "일치하는 노트가 없습니다" 메시지와 필터 초기화 버튼이 표시된다

### User Story 3 — 노트 카드 및 작성/수정 (Priority: P2)

노트 카드는 목록의 핵심 단위다. 제목, 날짜, 본문 미리보기, 태그, 편집/삭제 버튼이 카드 안에 있다. 편집 모드 진입 시 인라인으로 전환된다. 카드 간 간격, 타이포그래피 크기 위계, 버튼 위치가 일관되어야 한다.

**Why this priority**: 노트 CRUD가 핵심 기능이지만 인증/목록보다 개선 효과가 체감에서 한 단계 낮다.

**Independent Test**: 여러 노트가 있는 상태에서 한 카드를 열어 제목/본문/태그를 수정하고 저장. 카드 간 시각 일관성과 편집 전환이 부드러운지 확인

**Acceptance Scenarios**:

1. **Given** 노트 목록, **When** 카드 시각 검토, **Then** 모든 카드가 동일한 여백·폰트 크기·버튼 스타일을 가지고 시각적으로 균형 잡혀 있다
2. **Given** 편집 버튼 클릭, **When** 인라인 편집 모드 진입, **Then** 제목 입력, 본문 편집, 태그 입력 영역이 명확한 시각 구분으로 표시되고 저장/취소 버튼이 두드러진다
3. **Given** 마크다운 본문 포함 노트, **When** 뷰 모드 표시, **Then** 제목·코드블록·목록·이미지가 가독성 높은 타이포그래피로 렌더링된다
4. **Given** 노트 작성 영역(NoteComposer), **When** 시각 검토, **Then** 제목·본문·태그 입력 필드와 저장 버튼이 명확한 흐름으로 배치된다

### User Story 4 — 프로필/계정 화면 (Priority: P2)

프로필 화면에서 계정 정보 확인, 이름/표시이름 수정, 비밀번호 변경, Google 계정 연결이 가능하다. 현재 3열 그리드로 섹션이 나뉘어 있으며, 좁은 화면에서는 1열로 전환된다. 각 섹션이 독립적인 패널로 구분되어야 하고 연결 성공/실패 메시지가 명확해야 한다.

**Why this priority**: 사용자가 가끔 접근하는 화면이므로 노트 목록보다 우선순위가 낮다.

**Independent Test**: 프로필 화면에서 표시이름 수정 후 저장, Google 연결 섹션 확인

**Acceptance Scenarios**:

1. **Given** 프로필 화면 진입, **When** 레이아웃 확인, **Then** 계정 요약, 프로필 수정, 비밀번호 변경 섹션이 시각적으로 구분되고 각 섹션 제목이 명확하다
2. **Given** Google 계정 미연결 상태, **When** 연결 섹션 확인, **Then** Google 연결 버튼이 눈에 띄고 설명 문구가 간결하다
3. **Given** Google 계정 연결 성공, **When** 리다이렉트 후 프로필 화면, **Then** 성공 메시지가 초록색 등 긍정적 색상으로 표시되고 연결된 상태임을 아이콘/텍스트로 확인할 수 있다

---

### User Story 5 — 반응형 및 전체 레이아웃 일관성 (Priority: P3)

현재 880px 이하에서 그리드가 1열로 전환되는 미디어쿼리가 있다. 이 기준이 적절한지, 전환 시 각 요소가 자연스럽게 보이는지 확인하고 필요한 경우 개선한다.

**Why this priority**: 기능보다 polish에 해당하는 마지막 단계.

**Independent Test**: 브라우저 너비를 320px~1400px 사이에서 조절하며 레이아웃 깨짐 없이 전환되는지 확인

**Acceptance Scenarios**:

1. **Given** 880px 이하 너비, **When** 메인 화면 접근, **Then** 노트 작성란과 목록이 수직 정렬되고 서로 겹치거나 넘치지 않는다
2. **Given** 320px 최소 너비, **When** 각 화면 확인, **Then** 텍스트, 버튼, 입력 필드가 모두 잘려나가지 않고 스크롤 없이 읽힌다

---

### Edge Cases

- 매우 긴 노트 제목(60자+)을 가진 카드가 레이아웃을 깨지 않는가?
- 태그가 10개 이상인 노트의 태그 목록이 카드 레이아웃을 밀어내지 않는가?
- 에러 메시지와 성공 메시지가 동시에 표시되는 상황이 없는가?
- 이미지를 포함한 노트의 마크다운 렌더링이 카드 너비를 벗어나지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 모든 화면에서 색상, 간격, 타이포그래피, 버튼 스타일이 일관된 시각 언어를 따른다
- **FR-002**: 인증 화면은 앱 이름과 간단한 설명을 포함한 브랜드 영역을 폼 상단에 표시한다
- **FR-003**: 메인 화면 헤더는 현재 로그인한 사용자의 표시 이름을 보여준다
- **FR-004**: 노트 목록의 빈 상태, 로딩 상태, 오류 상태가 각각 시각적으로 구별되는 UI로 표시된다
- **FR-005**: 노트 카드는 본문 미리보기, 날짜, 태그가 일관된 타이포그래피 위계로 표시된다
- **FR-006**: 편집 모드와 뷰 모드 간 전환이 동일한 카드 영역 내에서 이루어지며 레이아웃 점프가 없다
- **FR-007**: 마크다운 렌더링 결과(제목, 코드블록, 목록, 이미지)가 카드 너비를 벗어나지 않는다
- **FR-008**: 프로필 화면의 각 섹션(계정 요약, 프로필 수정, 비밀번호 변경, Google 연결)이 패널로 명확히 구분된다
- **FR-009**: Google 연결 성공/실패 메시지가 색상으로 상태를 구별하고 자동으로 사라지거나 닫기가 가능하다
- **FR-010**: 880px 이하 너비에서 모든 주요 화면이 1열 세로 흐름으로 자연스럽게 전환된다
- **FR-011**: 색상 토큰, 간격 기준, 컴포넌트 스타일 규칙이 `app.css` 상단의 CSS 변수 섹션으로 정리되고 주석으로 목적이 명시된다
- **FR-012**: 레거시 마이그레이션이나 파일 기반 상태와 관련된 UI 문구나 상태 표시가 없다
- **FR-013**: 노트 본문 편집기는 "텍스트 편집기"(WYSIWYG)와 "마크다운" 두 모드를 제공하며, 각 모드는 편집/미리보기 서브탭을 가진다
- **FR-014**: 텍스트 편집기 모드에서 이미지 붙여넣기 및 파일 업로드로 이미지를 삽입할 수 있다
- **FR-015**: 마크다운 모드에서 이미지 붙여넣기 시 자동 업로드 후 `![image](url)` 문법으로 삽입된다
- **FR-016**: 각 노트 카드에서 전체화면 오버레이로 노트를 볼 수 있으며, ESC 키 또는 배경 클릭으로 닫힌다
- **FR-017**: 기존 마크다운 형식 노트와 새로운 HTML 형식 노트가 같은 목록에서 올바르게 렌더링된다

### Constitution Alignment *(mandatory)*

- **CA-001**: 모든 변경은 `frontend/src/` 내 컴포넌트와 스타일에만 적용된다. 백엔드 API 변경 없음. 프론트엔드와 백엔드의 HTTP API 경계는 유지된다.
- **CA-002**: 신규 API 엔드포인트 없음. 기존 엔드포인트 변경 없음. JSON envelope 형식 유지.
- **CA-003**: 신규 npm 패키지 없음. CSS 변수(`--color-*`, `--space-*` 등) 기반 디자인 토큰으로 기존 `app.css`를 확장하고, 기존 컴포넌트 수정으로만 구현한다. 유틸리티 클래스나 CSS-in-JS 방식은 사용하지 않는다.
- **CA-004**: 노트 편집은 기존 인라인 편집 방식을 유지한다. 모달 없음.
- **CA-005**: 스토리지 변경 없음. 레이아웃 프리퍼런스(localStorage)는 기존 훅 유지.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 인증 화면, 메인 화면, 프로필 화면에서 동일한 색상/버튼/간격 스타일이 사용되어 시각적으로 하나의 제품처럼 보인다 (검토자 육안 확인)
- **SC-002**: 빈 상태, 로딩 상태, 오류 상태 각각에 고유한 시각 표현이 존재하고 텍스트만으로 구별되지 않는다
- **SC-003**: 320px~1400px 너비 범위 전체에서 레이아웃 깨짐 없이 콘텐츠가 표시된다
- **SC-004**: 노트 카드가 10개 이상인 목록에서도 각 카드의 시각 경계가 명확하고 스캔하기 쉽다
- **SC-005**: 프론트엔드 기존 테스트 8/8이 변경 후에도 통과된다 (기능 회귀 없음)

## Assumptions

- 전체 디자인 톤: 현대적이고 깔끔한 방향. Notion/Linear 스타일처럼 흰 표면 + 중성 계열 텍스트 + 바이올렛 액센트
- 색상 팔레트: **zinc/slate 기반 + violet 액센트** (`#7c3aed`, `#0f172a`, `#f8fafc`). 기존 베이지·브라운 계열에서 완전 교체
- 폰트: IBM Plex Sans, Noto Sans KR — 유지
- 디자인 토큰은 CSS 변수(`--color-*`, `--space-*`) 방식으로 `app.css`에 정리한다
- 반응형은 주요 화면 깨짐 방지 수준을 우선한다. 기존 880px 브레이크포인트 유지
- 애니메이션/트랜지션은 최소화. hover·focus 등 기본 상호작용 피드백만 허용한다
- empty/loading/error 상태는 간결하고 일관된 시각 표현을 사용한다. 로딩: CSS 스피너 + 짧은 안내 텍스트. 빈 상태: 안내 문구 + 행동 유도. 오류: 에러 텍스트만. 과한 일러스트 없음.
- 다크모드 지원은 이번 범위에 포함되지 않는다
- 이미지 외 미디어 타입(동영상, 파일 첨부 등)은 범위 밖이다
- 기본 개발 및 검증 환경은 WSL Ubuntu + 로컬 브라우저다
- **보안**: 인증 토큰을 `sessionStorage`에 저장해 브라우저 종료 시 자동 만료됨. JWT 만료 시간 클라이언트 검증 포함
